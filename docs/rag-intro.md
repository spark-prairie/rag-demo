# RAG 技术简介

## 什么是 RAG
RAG (Retrieval-Augmented Generation, 检索增强生成) 是一种把外部知识库和大语言模型结合起来的方法.
它的核心思路是: 先从知识库里检索出和问题相关的片段, 再把这些片段作为上下文交给大模型作答.
这样可以让模型回答出训练语料里没有的内容, 并且答案更容易追溯来源.

## 为什么需要 RAG
大语言模型有三个常见问题:

1. **知识截止**: 模型训练完成后新出现的信息它不知道.
2. **幻觉**: 在不确定的时候模型倾向于编造看起来合理的答案.
3. **私有数据**: 企业内部文档, 个人笔记都不在公开语料里.

RAG 通过"先查再答"缓解了这三个问题 —— 知识在外挂的向量库里, 想更新就重新入库即可,
不需要重新训练模型.

## RAG 的核心组件
一个最小可用的 RAG 系统通常包含下面几块:

- **文档加载器**: 把 PDF / Markdown / 网页等原始文档读进来.
- **切分器 (splitter)**: 把长文档切成几百字一段的 chunk, 方便后续向量化和检索.
- **嵌入模型 (embedding model)**: 把文本映射成定长向量. 中文场景常用 BGE, M3E, GTE 等模型.
- **向量数据库**: 存储向量并支持相似度检索, 如 Chroma, FAISS, Milvus, Qdrant.
- **检索器 (retriever)**: 把用户问题向量化, 找出最相似的 top-k 个 chunk.
- **生成模型 (LLM)**: 把问题和检索到的 chunk 拼成 prompt, 让 LLM 输出最终答案.

## 常见的改进方向
- **混合检索 (hybrid search)**: 把向量检索和 BM25 关键字检索结合起来, 对专有名词更友好.
- **重排 (rerank)**: 对初步检索结果再用一个 cross-encoder 排一次, 提升 top-k 的精度.
- **查询改写 (query rewriting)**: 先让 LLM 把用户原始问题改写成更适合检索的形式.
- **父子切分 (parent-child chunking)**: 检索时用小 chunk 保证精度, 喂给 LLM 时用它所在的大段落保证上下文.
- **多轮对话**: 引入对话历史, 让它能回答"那刚才说的第二点展开讲讲"这种上下文相关的问题.

## 评估 RAG 的常见指标
- **召回率 (recall@k)**: top-k 的检索结果里是否包含标准答案来源.
- **忠实度 (faithfulness)**: 生成的答案是不是真的从检索片段里推出来的, 有没有幻觉.
- **答案相关性 (answer relevance)**: 答案是否真正回答了用户的问题.
- **上下文相关性 (context relevance)**: 检索到的片段是不是和问题相关, 有没有大量噪声.
